Painel — SQL Injection

Query SQL gerada pelo servidor

SELECT * FROM users WHERE username = '' AND password = ''
Aguardando entrada...
Como funciona

O servidor monta a query concatenando diretamente o input do usuário na string SQL.

Código vulnerável (PHP):
"SELECT * FROM users WHERE username='$user' AND password='$pass'"

Insira caracteres como ' ou -- para manipular a lógica da query.

Resumo da Aula — Segurança da Informação
Pilares da Segurança da Informação (CIA)
Confidencialidade
Apenas pessoas autorizadas acessam as informações.
Integridade
Dados não são alterados indevidamente.
Disponibilidade
Sistemas permanecem acessíveis quando necessário.
OWASP
SQL Injection historicamente no topo das vulnerabilidades críticas em apps web.
O que é SQL Injection
Ocorre quando o input do usuário é concatenado diretamente na query SQL, sem tratamento. O atacante altera a estrutura do comando originalmente planejado.
Código vulnerável
let sql = "SELECT * FROM users WHERE login = '" + login + "' AND senha = '" + senha + "'";
Consequências
  • Bypass de autenticação — entrar sem credenciais válidas
  • Acesso indevido a dados — ler registros sem permissão
  • Alteração / exclusão — modificar ou apagar dados do banco
  • Comprometimento total — em casos graves, controle do servidor
Análise lógica do ataque ' OR '1'='1
SELECT * FROM users WHERE login = 'admin' AND senha = '' OR '1'='1';
'1'='1' é sempre verdadeiro → OR verdadeiro → retorna todos os registros → bypass de autenticação.
Mitigação
  • Queries parametrizadas — separa estrutura SQL dos dados. Ex: WHERE login = ?
  • Validação server-side — proteção principal deve estar no backend
  • Menor privilégio — a aplicação usa só as permissões necessárias
  • Hash criptográfico de senhas — bcrypt, argon2 — nunca texto puro
  • Frameworks seguros — ORM, PreparedStatement, etc.
Vulnerável
Concatena entrada · Permite alteração da query · Alto risco
Seguro
Separa dado da estrutura · Preserva a query · Risco mitigado
Caso: XingLing Corporation
A empresa intermediadora de transações financeiras (Bradesco, Santander, BB…) sofreu fraude de centenas de milhões de reais via Engenharia Social.
Como aconteceu: um funcionário de baixo escalão foi corrompido financeiramente e inseriu um cavalo de troia via pendrive nas estações de trabalho. O malware realizou milhares de transferências de R$ 700.000,00 cada.
Problema crítico: a política de logs não conseguiu identificar o responsável — o único registro era a câmera de segurança física.
Nova Política de Segurança — Tópicos
  • 1. Segurança Física e Lógica — Como um analista júnior teve acesso irrestrito a tantas estações? Controle de acesso por cargo e área.
  • 2. Política de Logs — Registrar toda transação com identificação do operador, timestamp e origem. Logs devem ser auditáveis e à prova de adulteração.
  • 3. Política de Senhas e Criptografia — Implantação ou revisão imediata. Senhas fortes obrigatórias + criptografia de dados sensíveis.
  • 4. Sala Cofre — Diretivas adicionais de acesso físico, biometria, câmeras, controle duplo de entrada.
  • 5. Bloqueio de Dispositivos Externos — Nenhum recurso computacional deve permitir pendrive, CD, ou qualquer mídia externa não autorizada.
  • 6. ISO 27002-1 — Norma de referência para implantação da política de segurança da informação.
Lição Principal
Engenharia Social explora o fator humano — o elo mais fraco da segurança. Controles técnicos são necessários, mas insuficientes sem treinamento, conscientização e políticas de acesso adequadas ao perfil de cada colaborador.
Tipos de Ataque a Senhas
Força Bruta
Testa todas as combinações possíveis até encontrar a senha. Senhas curtas e simples são quebradas em segundos.
Dicionário
Usa listas de palavras comuns: senha123, admin, 123456, qwerty.
Engenharia Social
Explora dados pessoais: nome, data de nascimento, apelidos, nomes de animais de estimação.
Impacto
Quanto menor e mais simples a senha, mais rápido ela é descoberta por processo automatizado.
Senhas Fracas vs. Senhas Fortes
Senhas Fracas
123456   senha123   abc123   pedro2008   ifsp2024
Senha Forte — Exemplo
A7#kP1@xQ
Uma senha forte combina: MaiúsculasMinúsculasNúmerosEspeciais≥ 8 chars
Quanto maior a diversidade de caracteres e o comprimento, maior o espaço de combinações possíveis e mais difícil a descoberta.
Classificação de Força (exercício da aula)
  • Fraca → menos de 8 caracteres
  • Forte → entre 8 e 12 caracteres
  • Muito forte → mais de 12 caracteres
Critérios de pontuação (desafio)
  • +1 ponto: letra minúscula  |  +1: maiúscula  |  +1: número  |  +1: especial  |  +1: ≥8 chars
  • 0–2 pts = Fraca  |  3–4 pts = Média  |  5 pts = Forte
Papel do Desenvolvedor
  • Exigir tamanho mínimo e variedade de caracteres
  • Impedir senhas muito simples ou da lista negra
  • Oferecer gerador de senhas seguras
  • Classificar e exibir a força da senha em tempo real
  • Armazenar sempre com hash (bcrypt/argon2) — nunca texto puro
Projeto — Criptografia AES para Aula de Segurança
Experimento 01 — Navegador
AES diretamente no browser usando Web Crypto API. Sem dependência externa. Demonstra texto puro, cifrado, chave correta e chave errada.
Experimento 02 — MySQL
AES com MySQL + Stored Procedures + Node.js. A criptografia ocorre dentro do banco. Projeto sem MVC — foco em segurança.
Stored Procedures MySQL — AES + HEX
Os dados sensíveis são criptografados antes de serem gravados e decriptografados na consulta. A conversão HEX garante armazenamento seguro em colunas TEXT.
Gravação (INSERT)
INSERT INTO AGENDA_01(A01_nome, A01_telefone, A01_email) VALUES ( HEX(AES_ENCRYPT(V_A01_nome, V_chave)), HEX(AES_ENCRYPT(V_A01_telefone, V_chave)), HEX(AES_ENCRYPT(V_A01_email, V_chave)) );
Leitura (SELECT)
SELECT CAST(AES_DECRYPT(UNHEX(A01_NOME), 'chave') AS CHAR) Nome, CAST(AES_DECRYPT(UNHEX(A01_TELEFONE), 'chave') AS CHAR) Tel, CAST(AES_DECRYPT(UNHEX(A01_EMAIL), 'chave') AS CHAR) Email FROM AGENDA_01;
Procedures implementadas: Proc_InsAgendaProc_DelAgendaProc_AltAgendaProc_ConsAgenda
Problema Identificado — Tráfego em Cloud
Os dados percorrem a rede entre as estações clientes e o servidor de banco de dados em ambiente cloud. Mesmo com AES no banco, o tráfego pode ser interceptado se não houver criptografia na camada de transporte.
  • Risco: dados criptografados no banco, mas expostos durante o tráfego de rede
  • Solução de transporte: TLS/SSL obrigatório entre cliente e servidor MySQL
  • Risco da chave: chave AES compartilhada — se vazada, todos os dados são expostos
  • Solução de chave: HSM (Hardware Security Module) ou cofre de chaves (AWS KMS, HashiCorp Vault)
